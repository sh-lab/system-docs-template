
# エラー・例外処理方針

## 1. 基本方針
- 例外は原則として早期検知・早期通知を徹底する。
- 検査例外（Checked Exception）は業務例外、非検査例外（Unchecked Exception）はシステム例外として使い分ける。

## 2. 例外の分類
| 分類       | 説明                         | 例                       |
|------------|------------------------------|--------------------------|
| 業務例外   | 業務ルール違反（入力値不正は除く） | 在庫不足、重複申請、状態不整合等 |
※ 入力値不正（リクエストパラメータの未入力・形式不正等）は業務例外ではなく、コントローラ層でバリデーションエラーとして処理し、業務ロジック層には到達しない。
| システム例外 | システム障害や外部要因等     | DB障害、通信エラー等     |

## 3. 例外発生時の処理
- ログ出力（エラー内容・発生箇所・スタックトレース）
- ユーザー通知（業務例外は画面表示、システム例外は汎用エラー画面）
- 必要に応じてリトライやロールバックを実施

### 業務ロジック層での例外ハンドリング方針
- 原則として、例外発生時は個々の業務ロジックで何もせず、上位層（グローバル例外ハンドラ等）に委ねる。
- 局所的なリカバリやリトライが本当に必要な場合のみ、限定的にtry-catchを使用する。
- ログ出力（エラー内容・発生箇所・スタックトレース）
- ユーザー通知（業務例外は画面表示、システム例外は汎用エラー画面）
- 必要に応じてリトライやロールバックを実施

## 4. 例外クラス設計指針
- 業務例外：BusinessException等の基底クラスを継承
- システム例外：SystemException等の基底クラスを継承
- 命名規則：XXXException

## 5. 例外ハンドリング実装例
```java
try {
	// 業務処理
} catch (BusinessException ex) {
	
    // 例外発生時の業務処理
}
```

## 6. その他
- 外部API連携時の例外は、API種別ごとに個別ハンドリング方針を定める。
- UI層では業務例外のみユーザー通知、システム例外はシステム例外画面に遷移。
