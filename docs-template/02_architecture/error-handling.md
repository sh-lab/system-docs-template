# エラー処理方針

## 1. 基本方針
- エラーは**早期に検知し、責務を持つ層で明示的に扱う**。
- エラーを握りつぶさず、必要な文脈を保ったまま上位へ伝播させる。
- 実装方式は採用技術に従う。  
  （例：例外、Result型、エラー値、失敗レスポンスなど）
- ログ出力・ユーザー通知・ロールバック・リトライの責務を分離する。

## 2. エラーの分類
| 分類 | 説明 | 例 |
|------|------|----|
| 入力エラー | 入力値・形式・必須項目などの不正 | 未入力、型不正、形式不正 |
| 業務エラー | 業務ルールや状態整合性に反するもの | 在庫不足、重複申請、不正な状態遷移 |
| システムエラー | システム障害や外部要因による失敗 | DB障害、通信障害、タイムアウト |

- 入力エラーは**入口層**で扱い、業務ロジック層へ持ち込まない。
- 業務エラーは**想定内の失敗**として扱い、呼び出し元で適切に通知・応答する。
- システムエラーは**想定外または復旧判断が必要な失敗**として扱い、監視・調査可能な形で記録する。

## 3. 層ごとの扱い

### 3.1 入口層（Controller / Handler / API Endpoint）
- 入力値検証を行い、不正な入力はここで処理する。
- 下位層から返されたエラーを、UI / API の応答形式に変換する。
- エラー詳細をそのまま外部公開しない。

### 3.2 アプリケーション層
- ユースケース全体の失敗として扱う。
- 必要に応じてロールバック境界、リトライ方針、応答変換の起点を定める。
- 回復不能なエラーを局所的に握りつぶさない。

### 3.3 業務ロジック層
- 業務ルール違反は業務エラーとして表現する。
- その場で回復できないエラーは上位へ伝播させる。
- 局所的な代替処理やリトライは、必要性が明確な場合に限定する。

### 3.4 インフラ層
- DB、ファイル、外部APIなどの失敗を検知し、上位で扱える形に変換する。
- 技術詳細を隠蔽しつつ、原因追跡に必要な情報はログへ残す。

## 4. エラー発生時の処理方針
- **記録**：発生箇所、原因、入力条件、相関IDなどを記録する。
- **通知**：利用者・運用者それぞれに必要な粒度で通知する。
- **変換**：内部エラーをUI / API / バッチの表現に合わせて変換する。
- **回復**：リトライ、補償、ロールバックは明示的な方針がある場合のみ実施する。

## 5. 設計ルール
- エラー種別は、利用者に伝えるべきか・運用で調査すべきかで分ける。
- メッセージは**利用者向け**と**開発・運用向け**を分離する。
- エラーコードや分類子を持たせる場合は、呼び出し元で判定可能な粒度にする。
- 下位層の技術例外やライブラリエラーを、そのまま上位の業務判断に使わない。
- 成功と失敗の表現方法は、同一システム内でなるべく統一する。

## 6. テンプレート適用時の補足
- 具体的な実装規約（例外クラス設計、Result型の定義、HTTPステータス、ログ形式など）は、採用言語・フレームワークごとに別紙で定める。
- 外部連携のエラーは、連携先ごとに再試行可否・タイムアウト・フォールバック方針を定める。
- バッチ、非同期処理、イベント処理では、再実行時の重複実行対策もあわせて定める。
